# Book Analysis Modules

## 1. Classify the Book

Use source content, not category stereotypes.

```text
书籍类型：
判断依据：
当前章节更偏向：
本次采用的学习方法：
```

- Theory and argument: emphasizes claims, concepts, inference, evidence, interpretation, and scope.
- Technical and skill: emphasizes a problem, operational model, steps, practice, verification, and transfer.
- Mixed: combines both. Load only the module relevant to the current chapter.

If evidence is insufficient, state what additional material would permit classification.

## 2. Theory and Argument Books

For each important concept, analyze as needed:

- definition and why it is needed;
- phenomenon or problem it explains;
- distinction from nearby concepts;
- role in the book;
- applicable conditions and common misuse.

For each important claim, determine:

1. Claim.
2. Question answered.
3. Evidence or material used.
4. Inference from evidence to conclusion.
5. Hidden assumptions.
6. Evidence strength.
7. Applicable conditions.
8. Counterconditions or failures.
9. Likely misunderstanding.

Name the evidence design when possible. Separate cases and personal experience, cross-sectional surveys, correlations, experiments, longitudinal research, systematic reviews, meta-analyses, speculation, and value judgment.

For applications, state the scene, theoretical basis, action, conditions, likely misuse, and plausible alternative explanation.

## 3. Technical and Skill Books

For each important concept, analyze:

- the problem it solves;
- why existing tools or ideas are insufficient;
- how it works;
- how it differs from prior concepts;
- where it reappears later;
- prerequisites, scale limits, and maintenance/performance tradeoffs.

For an operation or program, trace:

- input;
- each state transition;
- output;
- creation versus mutation;
- side effects;
- likely failure points.

Ask the user to predict before explaining a result. Prefer a small fragment, framework, trace table, or minimal example over complete code.

Connect the chapter only to genuinely relevant projects. Explain the shared problem, what foundation the chapter supplies, where it appears in real work, and how deeply the user needs to understand it now.

Stop and offer `programming-learning-coach` when the task becomes an actual coding exercise, debugging session, test run, or independent-completion assessment.

## 4. Mixed Books

Apply argument analysis to explanatory claims and evidence. Apply technical analysis to methods and exercises. Do not mechanically fill every field.

Explicitly distinguish:

- a descriptive claim about how the world works;
- a normative judgment about what should be done;
- a practical procedure;
- evidence that the procedure works;
- the author's preference or teaching simplification.

## 5. Authorial Arrangement

When useful, examine:

1. Why this chapter appears here.
2. Required previous knowledge.
3. Later material it prepares.
4. Why the sequence may matter.
5. What breaks if the chapter is removed.
6. Capability the reader is expected to develop.

Label inferred teaching intentions as `【教学分析】`. Do not attribute intention to the author without explicit evidence.

## 6. Book Value and Limits

Assess only the dimensions supported by the book and, when permitted, external verification:

- problem solved within its field;
- distinguishing approach;
- stronger and weaker claims;
- limitations from era, culture, class, sample, or method;
- potentially outdated data or conclusions;
- important omissions;
- what the book helps a reader understand;
- what it cannot solve;
- concepts likely to be misused.

Do not force a fixed number of insights. During Phase 1 source reconstruction, label claims about the broader field or current validity as `【无法确认】` unless the book itself supplies the comparison. Use Phase 2 for external verification and Phase 3 for the user's final integration and transfer.
